# Phase 2 구현 노트

## 문서 성격

- 이 문서는 현재 코드 전체 설명서가 아니라, "Phase 2에서 실행기와 CSV 저장소 경계를 어떻게 열었는가"를 남긴 기록이다.
- 현재 프로젝트 최신 상태는 Phase 3 기준이다.
- 최신 사용법과 현재 지원 기능은 `README.md`, `Agents.md`, `PHASE3_IMPLEMENTATION_NOTES.md`를 우선해서 본다.

## 목적

- 이 문서는 내부 사고 과정을 그대로 적는 용도가 아닙니다.
- Phase 2에서 공유 가능한 설계 이유, 구현 순서, 리뷰 포인트를 정리합니다.
- 범위는 `execute_statement`, `storage_append_row`, CLI 실행 연결, 통합 테스트까지입니다.
- 파서 자체의 문법 확장은 이번 단계 범위에 포함하지 않습니다.

## Phase 2 목표

- Phase 1에서 만든 `Statement`를 실제 CSV 동작으로 연결한다.
- `INSERT`는 대상 CSV 파일에 새 행을 append 한다.
- `SELECT`는 대상 CSV 파일 내용을 헤더 포함 그대로 출력한다.
- 실행기와 저장소의 책임을 분리한 상태로 구현한다.
- 파서 테스트를 깨지 않으면서 실행 경로 테스트를 추가한다.

## 1. Phase 1 이후 상태 정리

Phase 1이 끝난 시점의 구조는 이미 다음 조건을 만족하고 있었다.

- SQL 파일 읽기
- 단일 SQL 문장 파싱
- `Statement` AST 구성
- 공통 에러 모델 제공
- 실행기와 저장소용 공개 API 자리 확보

즉, Phase 2의 핵심 작업은 “문법을 더 늘리는 것”이 아니라 “파싱 결과를 안전하게 실행으로 연결하는 것”이었다.

## 2. 이번 단계에서 먼저 고정한 규칙

구현 전에 아래 규칙을 먼저 확정했다.

- 범위는 `INSERT + SELECT`를 한 번에 끝낸다.
- CLI는 Phase 1의 AST 요약 대신 실행 결과를 출력한다.
- CSV 파일은 자동 생성하지 않는다.
- 헤더는 반드시 존재해야 한다.
- 헤더 컬럼 수와 `INSERT` 값 개수가 다르면 실패한다.
- 문자열 값은 CSV 저장 시 항상 큰따옴표로 감싼다.

이 규칙을 먼저 고정한 이유는, 실행기와 저장소 구현이 파일 포맷 계약에 강하게 묶이기 때문이다.

## 3. 왜 공개 인터페이스를 바꾸지 않았는가

기존 공개 인터페이스는 이미 충분했다.

- `parse_sql(const char *sql, Statement *out, SqlError *err)`
- `execute_statement(const Statement *stmt, const char *data_dir, FILE *out, SqlError *err)`
- `storage_append_row(...)`

이번 단계에서는 새 공개 타입이나 새 공개 함수보다, 이 세 인터페이스에 실제 구현을 채우는 편이 더 안전했다.

장점은 다음과 같다.

- Phase 1에서 학습한 구조를 그대로 유지할 수 있다.
- `main.c`는 parser -> executor 흐름만 연결하면 된다.
- 테스트도 기존 파서 테스트 위에 실행 테스트를 덧붙이기 쉽다.

## 4. 실행기와 저장소 책임 분리

### 실행기

실행기의 책임은 “무슨 동작을 해야 하는가”를 결정하는 것이다.

- `STMT_INSERT`면 저장소 append 호출
- `STMT_SELECT`면 대상 CSV를 열고 출력 스트림으로 복사
- 지원하지 않는 타입이면 실패

즉, 실행기는 AST를 보고 동작 경로를 선택하는 계층이다.

### 저장소

저장소의 책임은 “CSV 파일을 어떻게 다루는가”다.

- 테이블명 -> 파일 경로 변환
- 헤더 유효성 확인
- 헤더 컬럼 수 계산
- 새 CSV 행 직렬화
- append 시 줄 경계 보정

즉, 저장소는 파일 구조와 I/O 세부사항을 담당한다.

이 분리 덕분에 `src/execute.c`는 Statement 중심으로 읽히고, `src/storage.c`는 CSV 중심으로 읽힌다.

## 5. CSV 규칙을 어떻게 잡았는가

이번 구현에서 CSV는 학습용이지만, 최소한 데이터가 깨지지 않도록 규칙을 분명히 했다.

### 파일 매핑

- `table` -> `table.csv`
- `schema.table` -> `schema__table.csv`

이 규칙은 기존 메모 문서와 README에 이미 적혀 있었고, 구현에서도 그대로 사용했다.

### 헤더 검증

- 파일이 비어 있으면 실패
- 첫 줄이 비어 있으면 실패
- 헤더 개수와 `INSERT` 값 개수가 다르면 실패

이 검증은 “잘못된 CSV에 조용히 쓰지 않는다”는 쪽을 선택한 것이다.

### 문자열 직렬화

- 문자열은 항상 큰따옴표로 감싼다
- 내부 큰따옴표는 `""`로 바꾼다

이 규칙을 택한 이유는 다음과 같다.

- 쉼표가 있는 문자열을 안전하게 저장할 수 있다.
- 항상 같은 형식으로 저장되므로 디버깅이 쉽다.
- 학습자가 CSV escape 규칙을 눈으로 확인하기 좋다.

## 6. `INSERT` 경로 구현 포인트

`INSERT` 실행 흐름은 다음과 같다.

1. `Statement` 타입이 `STMT_INSERT`인지 확인
2. `data_dir`, `schema`, `table`로 대상 CSV 경로 계산
3. 파일을 열고 헤더를 읽어 컬럼 수 계산
4. `value_count`와 비교
5. 파일 끝이 개행으로 끝나지 않으면 개행 하나 보정
6. 새 CSV 행을 append
7. 성공 메시지 출력

중요한 점은 “CSV 직렬화는 저장소가 담당하고, 실행기는 성공 메시지만 담당한다”는 것이다.

## 7. `SELECT` 경로 구현 포인트

`SELECT`는 Phase 2에서도 일부러 단순하게 유지했다.

흐름은 다음과 같다.

1. `Statement` 타입이 `STMT_SELECT`인지 확인
2. 대상 CSV 경로 계산
3. 파일 존재와 헤더 존재를 검증
4. 파일 전체를 출력 스트림으로 복사

즉, 현재 `SELECT`는 필터링이나 컬럼 선택이 아니라 “CSV 전체 조회”에 가깝다.

이 단순함이 중요한 이유는, 아직은 저장 형식과 실행 경계 학습이 핵심이기 때문이다.

## 8. 왜 CLI를 요약 출력에서 실행 출력으로 바꿨는가

Phase 1에서는 파서 확인이 목표였으므로 AST 요약이 유용했다.

하지만 Phase 2에서는 성공 기준이 달라졌다.

- `INSERT`가 실제로 파일에 반영되는가
- `SELECT`가 실제 CSV를 출력하는가

그래서 `main.c`는 다음 흐름으로 바뀌었다.

1. SQL 파일 읽기
2. `parse_sql`
3. `execute_statement`
4. 실패 시 parse/execute 에러를 각각 출력

즉, CLI의 관찰 포인트가 “파싱 결과”에서 “실행 결과”로 이동했다.

## 9. 에러 처리에서 유지한 원칙

이번 단계에서도 Phase 1의 에러 전달 원칙을 그대로 유지했다.

- 성공/실패는 `int` 반환값으로 구분
- 자세한 원인은 `SqlError`에 기록
- 실패 메시지는 가능한 한 파일 경로와 원인을 같이 담음

예를 들면 아래 종류의 실패를 구분한다.

- CSV 파일 열기 실패
- 헤더 없음
- 헤더 개수와 값 개수 불일치
- SELECT 출력 쓰기 실패
- 지원하지 않는 Statement 타입

## 10. 테스트 전략

기존 `tests/test_parser.c`는 parser 전용이었지만, 이번 단계에서는 실행 테스트도 같은 실행 파일에 합쳤다.

이 방식을 택한 이유는 다음과 같다.

- 현재 프로젝트 규모에서는 단일 테스트 바이너리가 단순하다.
- `make test` 기준을 유지하기 쉽다.
- Phase 1 회귀와 Phase 2 검증을 한 번에 확인할 수 있다.

### 유지한 파서 테스트

- 단순 INSERT
- 단순 SELECT
- schema 포함 이름
- 음수 정수
- 세미콜론 누락
- 컬럼 목록 사용
- 닫히지 않은 문자열
- trailing comma
- 다중 문장
- WHERE 절

### 추가한 실행 테스트

- INSERT 성공 후 CSV row append 확인
- 대상 CSV가 없을 때 실패 확인
- 헤더 개수와 값 개수 불일치 실패 확인
- schema 경로 매핑 확인
- 큰따옴표 escape 확인
- SELECT 출력 확인
- 빈 CSV 실패 확인
- INSERT 후 SELECT round-trip 확인

## 11. 리뷰 포인트

코드리뷰에서 특히 보면 좋은 포인트는 아래와 같다.

### 실행기

- `Statement.type` 분기가 명확한가
- `INSERT`와 `SELECT`가 서로의 책임을 침범하지 않는가
- 출력 스트림 오류를 제대로 처리하는가

### 저장소

- 파일 경로 계산이 규칙과 일치하는가
- 헤더가 없는 파일을 조용히 통과시키지 않는가
- 문자열 직렬화가 CSV를 깨뜨리지 않는가
- append 전에 줄 경계를 안전하게 맞추는가

### 테스트

- 파서 회귀와 실행 회귀를 동시에 막고 있는가
- 임시 디렉터리를 사용해 tracked data를 직접 오염시키지 않는가

## 12. 이번 단계에서 의도적으로 뺀 기능

- CSV 파일 자동 생성
- 컬럼 이름 기반 매핑
- 타입 검증 강화
- `WHERE`
- 특정 컬럼만 조회
- 여러 SQL 문장 실행
- 더 넓은 SQL 문자열 escape 문법

즉, Phase 2는 “CSV 실행 경로를 안정적으로 열기”까지를 목표로 하고, 데이터베이스 기능 확장은 다음 단계로 남겨 두었다.

현재 기준으로는 이 항목들 중 다수가 Phase 3에서 실제로 추가되었다.

- CSV 파일 자동 생성
- 컬럼 이름 기반 매핑
- 타입 검증 강화
- 특정 컬럼만 조회
- 여러 SQL 문장 실행
- SQL 표준 `''` 문자열 escape

## 13. 정리

Phase 2의 핵심은 새로운 문법을 더하는 것이 아니라, 이미 만든 AST를 실제 동작으로 연결하는 것이었다.

이번 구현으로 프로젝트는 다음 흐름을 갖게 되었다.

1. SQL 파일 읽기
2. 수동 스캐너 기반 파싱
3. `Statement` AST 생성
4. 실행기 분기
5. CSV 저장 또는 조회

그래서 이제 이 프로젝트는 “파서를 배운다”에서 끝나지 않고, “파싱 결과가 실제 파일 동작으로 이어지는 구조”까지 학습할 수 있는 상태가 되었다.

## 14. 당시 상태와 현재 상태

### Phase 2 당시 상태

- 단일 문장 실행만 지원했다.
- `SELECT *`만 허용했다.
- CSV는 미리 존재해야 했다.
- schema 파일은 아직 없었다.

### 현재 상태

- 현재 프로젝트는 Phase 3까지 구현되었다.
- `parse_sql_script`, `execute_script`, schema-aware storage, staging/rollback이 추가되었다.
- 그래서 이 문서는 "Phase 3 이전에 무엇이 단순화되어 있었는가"를 이해하는 용도로 읽는 것이 적절하다.
