# 프로젝트 예상 면접 질문 10선

이 문서는 `SQL_WednsdayCodingClub` 프로젝트를 기준으로, 실제 면접이나 코드 리뷰 자리에서 나올 가능성이 높은 질문 10개를 정리한 자료입니다.  
각 질문 아래에는 답변할 때 꼭 짚으면 좋은 핵심 포인트를 짧게 붙였습니다.

## 1. 이 프로젝트의 전체 아키텍처를 어떻게 설명할 수 있나요?

### 답변 포인트

- 크게 `parser`, `execute`, `storage` 세 계층으로 나눴습니다.
- `parser`는 SQL 문법을 읽어 `Statement` / `SqlScript` AST를 만듭니다.
- `execute`는 AST를 보고 어떤 동작을 해야 하는지 결정합니다.
- `storage`는 실제 CSV/schema 파일 I/O를 담당합니다.
- 이렇게 나누면 문법 해석, 실행 정책, 파일 저장 책임이 섞이지 않아 학습과 확장이 쉬워집니다.

## 2. SQL 파서를 `strtok` 같은 방식이 아니라 수동 스캐너로 구현한 이유는 무엇인가요?

### 답변 포인트

- SQL은 공백, 키워드, 구분자, 문자열 리터럴, 에러 위치 추적이 중요합니다.
- `strtok`는 입력 문자열을 파괴하고, 위치 추적과 quoted string 처리에 약합니다.
- 수동 스캐너는 `input`, `length`, `pos`를 기반으로 현재 위치를 정확히 관리할 수 있습니다.
- 덕분에 `err.position` 같은 디버깅 정보도 정확하게 전달할 수 있습니다.

## 3. `Statement`와 `SqlScript` 구조를 왜 이렇게 설계했나요?

### 답변 포인트

- `Statement`는 문장 하나를 표현하고, `SqlScript`는 여러 문장을 순서대로 담습니다.
- `Statement` 안에는 `schema`, `table`, `columns`, `values`, `select_all` 같은 최소 실행 정보를 넣었습니다.
- `INSERT`와 `SELECT`가 같은 구조를 공유하되, 필요한 필드만 사용하도록 설계했습니다.
- 이렇게 해야 파서, 실행기, 테스트가 같은 AST 계약을 기준으로 움직일 수 있습니다.

## 4. 메모리 소유권과 해제 전략은 어떻게 가져갔나요?

### 답변 포인트

- 파서가 AST 내부 문자열과 배열을 모두 heap에 구성합니다.
- 성공 시 소유권은 호출자에게 넘어가고, 호출자는 `statement_free`, `sql_script_free`로 정리합니다.
- 실패 시 부분적으로 할당된 메모리는 내부에서 정리하도록 했습니다.
- C 프로젝트에서는 이 ownership 규칙이 흔들리면 바로 leak나 double free가 나기 때문에 가장 먼저 고정한 계약입니다.

## 5. 왜 CSV만 두지 않고 `.schema.csv` 파일을 따로 둔 건가요?

### 답변 포인트

- CSV 헤더만으로는 타입 정보를 알 수 없습니다.
- Phase 3에서 컬럼 이름 기반 매핑, 타입 검증, projection SELECT를 하려면 schema 정보가 필요했습니다.
- 그래서 `name,type` 형식의 별도 schema 파일을 두고 `INT`, `STRING` 타입만 지원하게 했습니다.
- 이 구조 덕분에 CSV는 데이터 저장용, schema는 계약 검증용으로 역할이 분리됩니다.

## 6. 컬럼 이름 기반 INSERT는 내부적으로 어떻게 동작하나요?

### 답변 포인트

- 예를 들어 `INSERT INTO users (name, id, age) VALUES (...)`처럼 들어오면, 입력 순서 그대로 저장하지 않습니다.
- 먼저 schema를 읽고 컬럼 이름을 schema index로 매핑합니다.
- 그 다음 값을 schema 순서로 재배열한 뒤 타입 검증을 하고 CSV에 저장합니다.
- 즉, SQL 입력 순서가 아니라 schema 기준 저장 순서를 보장하는 것이 핵심입니다.

## 7. `SELECT name, age FROM users` 같은 특정 컬럼 조회는 왜 schema가 있을 때만 허용하나요?

### 답변 포인트

- projection SELECT는 컬럼 이름을 실제 CSV column index로 바꿔야 합니다.
- schema가 없으면 헤더 문자열만 믿고 동작해야 하는데, 타입/순서 계약이 약해집니다.
- 이 프로젝트는 학습용이지만 계약이 불명확한 동작은 줄이기 위해 schema 기반으로만 projection을 열었습니다.
- 대신 schema가 없는 legacy 테이블은 `SELECT *`와 positional INSERT만 허용합니다.

## 8. 여러 SQL 문장을 실행할 때 `all-or-nothing`을 어떻게 보장했나요?

### 답변 포인트

- 원본 `data_dir`를 바로 수정하지 않고, 먼저 stage 디렉터리에서 실행합니다.
- 필요한 CSV/schema를 stage로 복사하고, 모든 문장을 stage에 반영합니다.
- 전부 성공하면 마지막에 stage 결과를 원본으로 commit 합니다.
- 중간에 하나라도 실패하면 stage를 버리기 때문에 rollback 효과가 납니다.
- 출력도 임시 버퍼에 모았다가 성공 후에만 `stdout`으로 내보냅니다.

## 9. 이 프로젝트에서 에러 처리는 어떤 방식으로 설계했나요?

### 답변 포인트

- C에서는 예외가 없기 때문에 `int` 성공/실패 코드와 `SqlError` 구조체를 같이 씁니다.
- `SqlError`에는 `code`, `position`, `message`를 담습니다.
- 파싱 오류, I/O 오류, 인자 오류, 메모리 오류를 분리해서 다룹니다.
- 면접에서는 “실패 원인을 호출자가 분기할 수 있게 했다”는 점을 강조하면 좋습니다.

## 10. 이 프로젝트를 더 확장한다면 다음 우선순위는 무엇인가요?

### 답변 포인트

- `WHERE`, `NULL`, 부분 컬럼 INSERT, 더 많은 타입 지원이 먼저입니다.
- 그 다음은 CSV 기반 한계를 줄이기 위한 인덱싱 또는 row abstraction을 고민할 수 있습니다.
- parser와 storage가 이미 분리돼 있어서, 기능 추가는 가능하지만 CSV 자체가 가진 한계는 분명합니다.
- 그래서 “현재 구조는 학습용 mini DB로는 좋지만, 실제 DB 엔진 대체용은 아니다”라고 선을 분명히 그어주는 게 좋습니다.

## 면접에서 특히 자주 꼬리질문 나오는 주제

- 왜 CSV를 선택했는가
- 왜 schema를 따로 두었는가
- rollback을 진짜 어떻게 보장하는가
- C에서 메모리 안전성을 어떻게 챙겼는가
- `realloc` 크기 계산 overflow는 어떻게 막았는가
- 테스트가 어떤 회귀를 잡아주는가

## 빠른 답변 팁

- 구조 질문이 나오면 `parser -> execute -> storage` 순서로 답합니다.
- 기능 질문이 나오면 “입력 계약 -> 내부 처리 -> 실패 조건” 순서로 답합니다.
- 트레이드오프 질문이 나오면 “학습용 단순성 vs 실제 DB 기능성”으로 정리하면 흔들리지 않습니다.

# 기술 개념 예상 질문 10선

아래 질문들은 이 프로젝트를 계기로 자주 이어지는 기술 개념 질문들입니다.  
즉, 프로젝트 구현 디테일을 물은 뒤 바로 이어서 나올 수 있는 CS/시스템/언어 개념 질문들입니다.

## 1. AST(Abstract Syntax Tree)는 무엇이고 왜 필요한가요?

### 답변 포인트

- AST는 파싱된 문법 구조를 트리나 구조체 형태로 표현한 결과입니다.
- 원본 문자열을 바로 실행하지 않고, 의미 있는 중간 표현으로 바꾸기 위해 필요합니다.
- 이 프로젝트에서는 `Statement`와 `SqlScript`가 AST 역할을 합니다.
- AST가 있으면 파싱과 실행을 분리할 수 있습니다.

## 2. Lexer와 Parser의 차이는 무엇인가요?

### 답변 포인트

- Lexer는 문자열을 토큰 단위로 쪼개는 단계입니다.
- Parser는 그 토큰이나 문자 흐름을 문법 구조로 해석하는 단계입니다.
- 이 프로젝트는 별도 lexer를 두지 않고 parser 내부에서 수동 스캐너로 둘을 함께 처리합니다.
- 작은 프로젝트에서는 이 방식이 단순하지만, 문법이 커지면 lexer/parser 분리가 유리해집니다.

## 3. Tagged Union이 무엇이고 C에서 왜 자주 쓰이나요?

### 답변 포인트

- 여러 타입 중 하나를 저장해야 할 때 `enum + union` 조합으로 표현하는 방식입니다.
- `SqlValue`에서 `type`이 현재 값 종류를 나타내고, union이 실제 값을 담습니다.
- C에는 sum type이 없기 때문에 이런 방식이 실용적입니다.
- 메모리를 절약하면서도 타입 구분이 가능합니다.

## 4. `malloc`, `calloc`, `realloc`의 차이는 무엇인가요?

### 답변 포인트

- `malloc`은 지정한 크기만큼 메모리를 할당합니다.
- `calloc`은 여러 개를 할당하면서 0으로 초기화합니다.
- `realloc`은 기존 메모리 블록 크기를 늘리거나 줄입니다.
- `realloc`은 실패 시 원본 포인터가 그대로 남을 수 있어서 바로 덮어쓰면 안 됩니다.

## 5. `realloc`을 쓸 때 정수 오버플로를 왜 조심해야 하나요?

### 답변 포인트

- 보통 `count * sizeof(T)`로 새 크기를 계산하는데, 이 곱셈이 overflow 되면 실제보다 작은 메모리가 할당될 수 있습니다.
- 그러면 뒤에서 배열에 값을 쓰는 순간 메모리 손상이 날 수 있습니다.
- 그래서 `SIZE_MAX / sizeof(T)` 기준으로 미리 가드를 넣는 게 중요합니다.
- 이번 프로젝트도 이 부분이 PR 리뷰 포인트였습니다.

## 6. 얕은 복사와 깊은 복사의 차이는 무엇인가요?

### 답변 포인트

- 얕은 복사는 포인터 값만 복사해서 같은 메모리를 같이 참조합니다.
- 깊은 복사는 실제 데이터까지 새 메모리에 복사합니다.
- 파서에서 문자열 리터럴이나 식별자를 그냥 가리키게 두면 원본 버퍼 수명에 묶입니다.
- 그래서 이 프로젝트는 AST가 원본 SQL 문자열과 독립적으로 살 수 있게 깊은 복사를 사용합니다.

## 7. 트랜잭션의 atomicity는 무엇인가요?

### 답변 포인트

- atomicity는 “전부 성공하거나 전부 실패해야 한다”는 성질입니다.
- DB에서는 중간 상태가 외부에 보이면 안 됩니다.
- 이 프로젝트는 실제 DB는 아니지만, stage 디렉터리와 commit/rollback으로 그 개념을 흉내 냈습니다.
- multi-statement 실행에서 가장 중요한 개념입니다.

## 8. 버퍼링(buffering)은 왜 필요한가요?

### 답변 포인트

- 출력이나 파일 쓰기를 바로바로 외부에 내보내지 않고 임시로 모아두는 방식입니다.
- 성능 이유도 있지만, 이 프로젝트에서는 “실패한 실행 결과를 콘솔에 남기지 않기 위해”도 필요했습니다.
- 그래서 `SELECT` 결과도 임시 파일에 모았다가 전부 성공한 뒤에만 출력합니다.
- 데이터 rollback과 출력 rollback을 같이 맞추는 데 중요합니다.

## 9. CSV에서 문자열 quoting과 escaping이 왜 필요한가요?

### 답변 포인트

- CSV는 쉼표와 줄바꿈이 필드 구분에도 쓰이기 때문에 문자열 안에 이런 문자가 들어오면 구분이 깨질 수 있습니다.
- 그래서 문자열은 큰따옴표로 감싸고, 내부 큰따옴표는 `""`로 escape 합니다.
- 이 규칙을 지키지 않으면 읽을 때 컬럼 수가 틀어지거나 데이터가 깨집니다.
- 저장과 조회 양쪽 모두 같은 규칙을 따라야 합니다.

## 10. POSIX/Windows 차이를 고려한 portable C 코드는 어떤 점을 신경 써야 하나요?

### 답변 포인트

- 경로 구분자, 디렉터리 생성 함수, PID 함수, 콘솔 인코딩 같은 부분이 다릅니다.
- 이 프로젝트도 `_mkdir` / `mkdir`, `_getpid` / `getpid`, Windows UTF-8 콘솔 설정 같은 분기를 뒀습니다.
- 파일 I/O 자체는 가능한 한 표준 C 함수로 유지하고, OS 의존 부분만 조건부 컴파일로 분리하는 것이 좋습니다.
- 면접에서는 “표준 C로 최대한 유지하고, OS 차이는 가장 얇게 감쌌다”고 설명하면 좋습니다.

## 기술 개념 질문 답변 팁

- 정의만 말하지 말고, 프로젝트의 실제 예시와 연결해서 답합니다.
- 예를 들어 `atomicity`를 물으면 바로 `stage 디렉터리` 사례를 같이 말합니다.
- `Tagged Union`, `realloc overflow`, `deep copy`는 `SqlValue`, AST, parser 메모리 관리와 연결해서 답하면 훨씬 설득력이 있습니다.
